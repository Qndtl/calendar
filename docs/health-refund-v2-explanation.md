# 신규 건강보험 환급 v2 — 단계별 설명

---

## 사전 개념

### 월 표기 기준 (환급 계산 기준월 기준)

| 변수명 | 실제 월 |
|---|---|
| sorted4 | 4개월 전 |
| sorted3 | 3개월 전 |
| sorted2 | 2개월 전 |

> 예: 기준월이 5월이면 → sorted4 = 2월, sorted3 = 3월, sorted2 = 4월

---

### 핵심 개념: "기간"이란?

기존 로직은 첫 출역일로부터 고정 30일을 더해 윈도우를 계산했지만,  
v2는 **"기간"** 이라는 개념을 도입해 더 직관적으로 정의합니다.

**4개월전 기간:**
- 시작: 4개월전 첫 출역일
- 끝: 첫 출역일 + (해당 월 일수 - 1)일
  - 단, 첫 출역일이 초일(1일)이면 → 해당 월 말일까지
- 예) 4개월전 첫 출역이 2월 10일(2월=28일) → 기간은 2/10 ~ 3/9
- 예) 4개월전 첫 출역이 2월 1일 → 기간은 2/1 ~ 2/28

**3개월전 기간 (from-start):**
- 시작: 3개월전 첫 출역일
- 끝: 첫 출역일 + (해당 월 일수 - 1)일
- 예) 3개월전 첫 출역이 3월 5일(3월=31일) → 기간은 3/5 ~ 4/4
- 예) 3개월전 첫 출역이 4월 5일(4월=30일) → 기간은 4/5 ~ 5/4

**기간 내 출역 수:**  
4개월전 + 3개월전 + 2개월전 전체 날짜(`allDates`) 중 해당 기간 범위 안에 들어오는 날 수

---

## 단계별 흐름

---

### Step 1: 합산 출역이 8일 미만인가?

합산 기준은 4개월전(sorted4) 유무에 따라 다르다:

- **4개월전 없음(sorted4 = 0)**: 3개월전 + 2개월전 합산  
  → 3개월전 기간(period3)이 2개월전 월까지 걸칠 수 있으므로 2개월전도 포함
- **4개월전 있음**: 4개월전 + 3개월전 합산

이 합계가 8일 미만이면 공제 요건 자체가 성립되지 않으므로 → **환급**  
(단, 공제된 내역이 있을 때만 환급 대상으로 처리)

합계가 8일 이상이면 Step 2로 진행한다.

---

### Step 2: 기간 계산

4개월전 기간과 3개월전 기간을 각각 계산한다.  
각 기간 내 출역 수(`period4Count`, `period3Count`)도 함께 산출한다.

기간 계산이 끝나면 Step 3으로 진행한다.

---

### Step 3-4: 모든 기간이 8일 미만인가?

4개월전 기간, 3개월전 기간 모두 8일 미만이면  
어느 쪽 기간도 공제 요건을 충족하지 못한 것이므로 → **환급**

하나라도 8일 이상이면 Step 5로 진행한다.

---

### Step 5a: 4개월전 기간이 8일 이상인가?

4개월전 기간의 출역이 8일 이상이면, 공제가 정당한지 추가로 확인한다.

다음 두 조건 중 하나라도 해당되면 → **공제 정당**
- 4개월전 기간이 끝난 이후 날짜(2개월전까지 포함)에 출역이 있다
- 기간의 첫날 AND 마지막날 모두 출역이 있다 (기간이 실제로 꽉 차 있음을 의미)

공제 정당으로 판단된 이후:
- 공제내역이 이미 있으면 → **금액 비교** (이미 낸 공제금액과 비교 후 정산)
- 공제내역이 없으면 → **징수** (공제가 됐어야 했는데 안 됐으므로 추가 징수)
  - 3개월전 출역 날짜 전체가 징수 대상

두 조건 모두 해당 없으면 → **환급**  
(기간이 8일 이상이지만 기간 전후로 고용이 이어지지 않고, 경계도 불명확한 경우)

---

### Step 5b: 4개월전 기간이 8일 미만일 때

다음 순서로 "앞에서 온 공제 정당 신호"를 확인한다:

**① 5개월전 출역 있음(sorted5 > 0)**  
이전 크로스월 고용이 이미 성립 → 공제 정당
- 3개월전 >= 8일 → **금액 비교 / 징수**
- 3개월전 < 8일 → **환급**

**② 4개월전 초일 출역**  
연속 고용 신호 → 공제 정당
- 3개월전 >= 8일 → **금액 비교 / 징수**
- 3개월전 < 8일 → **환급**

**③ 4개월전 있음 + 3개월전 단독 8일 이상 (Branch F)**  
3개월전이 이미 단독으로 공제 요건 충족 → 공제 정당 → **금액 비교 / 징수**

**④ 4개월전 있음 + 3개월전 단독 8일 미만 (Branch F sub)** → **환급**  
4개월전 기간도 부족하고 3개월전도 부족.  
period3 window에 2개월전 날짜가 포함되더라도 이 경우는 요건 불충족으로 본다.

**⑤ ①~④ 모두 해당 없음 (sorted4 = 0)**  
→ 3개월전 기간 체크(Step 5c/5d)로 진행

---

### Step 5c: 3개월전 기간이 8일 이상인가?

> Step 5b의 ①~④에 모두 해당하지 않는 경우, 즉 **sorted4 = 0(4개월전 없음)** 일 때만 이 단계에 도달한다.

period3Count(3개월전 기간 내 출역 수)가 8일 이상인지 확인한다.  
기간(period3)은 sorted3[0] ~ sorted3[0]+(해당월 일수-1)이며, 2개월전 날짜가 포함될 수 있다.

기간 내 출역이 8일 이상이고, 다음 조건을 충족하면 → **공제 정당**
- **기간 종료일 이후 출역**: 기간 종료일(period3End) 이후에 출역이 있다
- 또는 **3개월전이 1/30, 1/31 특수일**이면서 2개월전 말일 출역

공제 정당으로 판단된 이후:
- 공제내역이 이미 있으면 → **금액 비교**
- 공제내역이 없으면 → **징수** (3개월전 출역 날짜 전체가 징수 대상)

조건을 충족하지 않으면 → **환급**

---

### Step 5d: 3개월전 기간이 8일 미만이면

기간 내 출역이 8 미만이면 → 무조건 **환급**

---

## 결과 요약

| 상황 | 공제내역 있음 | 공제내역 없음 |
|---|---|---|
| 합산 < 8 | 환급 | 비대상 |
| 모든 기간 < 8 | 환급 | 비대상 |
| 4개월전 기간 >= 8, 공제 정당 | 금액 비교 | **징수** |
| 4개월전 기간 >= 8, 공제 부당 | 환급 | 비대상 |
| 5개월전/4개월전 있음(sorted3>=8) — Branch F | 금액 비교 | **징수** |
| 5개월전/4개월전 있음(sorted3<8) | 환급 | 비대상 |
| sorted4=0, 3개월전 기간 >= 8, 공제 정당 | 금액 비교 | **징수** |
| sorted4=0, 3개월전 기간 >= 8, 공제 부당 | 환급 | 비대상 |
| sorted4=0, 3개월전 기간 < 8 | 환급 | 비대상 |

## 기존 로직과의 핵심 차이

기존 로직은 4개월전 유무, 초일 출역 여부, 1/30·31 특수일 등을 **개별 분기**로 처리해  
분기가 매우 많았다. v2는 이를 **"기간"** 개념으로 추상화해 처리한다.

- 기존: 4개월전 없음 → 3개월전 초일 출역? → 3개월전 말일 출역? → ...
- v2: 기간 계산 → 기간 내 출역 수 비교 → 기간 전후 출역 확인

결과적으로 동일한 케이스를 더 적은 분기로 커버하며,  
기존에 버그로 지적됐던 `+31일 이후 범위`(2개월전까지 보지 않던 문제)도 자연스럽게 해결된다.

**Step 1 합산 공식 (sorted4=0 vs sorted4>0)**  
4개월전이 없으면 3개월전 기간이 2개월전까지 걸칠 수 있으므로 sorted3+sorted2로 합산한다.  
반면 4개월전이 있으면 sorted4+sorted3으로 합산한다.  
이로써 sorted4=0인 케이스(기존 A-3 분기)도 period3 window로 올바르게 처리된다.

**Branch F sub (sorted4>0 + sorted3<8 → 환급)**  
sorted4가 있어도 period4Count<8이고 sorted3<8이면,  
period3 window에 sorted2 날짜가 포함되어 period3Count>=8이 되더라도 환급으로 처리한다.  
이는 기존 F-2 분기(filtered<8 + sorted3<8 → 환급)와 동일한 결과를 보장한다.
